Lesson 1: The Big Picture

What iniLive Studio is, how it works end-to-end, and why the milestones are ordered the way they are.
Estimated time: 10 minutes reading + quiz

What you'll learn

By the end of this lesson, you'll be able to trace a live broadcast from a Host's browser all the way to a viewer on YouTube — naming every major system component along the way. You'll also understand why the plan proves the risky parts first.

The Product in One Paragraph

iniLive Studio is a browser-based production room for streaming a small group conversation to YouTube. A signed-in Host creates a reusable Room, invites up to three Guests by sharing a link, and controls when the Room is broadcast. Guests join without accounts. The Audience watches on YouTube — they never enter the product itself.

Key mental model Think of it as a private backstage (the Room) with a one-way window to YouTube (the Broadcast). The Host decides when to open and close that window.

Let's walk through the five key components:

1. WebRTC SFU — The Media Router

Every person in the Room sends their camera/mic feed to a Selective Forwarding Unit (SFU). The SFU doesn't mix or compose anything — it just routes each person's individual media stream to everyone else. Think of it as a smart switchboard: each participant uploads their stream once, and the SFU forwards copies to the other participants.

iniLive Studio will use a managed SFU service (the plan spike inLive first, then LiveKit Cloud as fallback). This means you don't run the SFU yourself — you use their APIs to create rooms and get tokens.

Why an SFU instead of peer-to-peer? With 4 people in a peer-to-peer mesh, each person would upload their video 3 times and download 3 streams. An SFU lets each person upload once and download 3, cutting upload bandwidth by 3×. This matters on consumer connections.

2. Canvas Composer — Browser-Side Composition

This is a key architectural choice: the Host's browser takes all the individual video streams from the SFU and draws them onto a single HTML <canvas> element at 720p/30fps. This canvas includes the participant grid layout, display name labels, and camera-off placeholders.

The canvas then exposes its content as a MediaStream via captureStream(). This is the Composed Room Feed — one single video representing the entire Room.

Risk to watch Browser composition is CPU-intensive. Drawing 4 video streams + a screen share onto a canvas at 30fps is real work. The plan's exit criteria require ≥28fps on an M1 Mac. If this becomes a bottleneck, it's the hardest thing to change later.

3. WHIP Client — Sending Composed Video to the Server

WHIP (WebRTC-HTTP Ingestion Protocol) is how the browser sends the Composed Room Feed to the server. It's essentially "WebRTC, but for sending media to a server instead of between peers." The Host browser is a WHIP client; the Broadcast Bridge is a WHIP server.

Authentication uses a per-Broadcast bearer token — each time the Host starts a broadcast, the SvelteKit backend issues a fresh token that the WHIP client includes in its request.

4. Broadcast Bridge — Server-Side Transcoding

The Broadcast Bridge is a GStreamer pipeline running on the VPS. It receives the Composed Room Feed via WHIP (WebRTC), transcodes from VP8/VP9 to H.264, encodes audio as AAC, and pushes the result to YouTube via RTMP.

YouTube requires H.264 + AAC over RTMP. Browsers send VP8 or VP9 over WebRTC. The bridge is the translator between these two worlds.

5. YouTube — The Audience's Window

The Audience watches on YouTube. There's no watch page in iniLive Studio itself. The Host pastes their YouTube RTMP URL and stream key each time they broadcast — credentials are ephemeral and never stored.

Why This Milestone Order?

The plan has three milestones that build on each other in a specific order. The order is deliberate — it proves the riskiest unknowns first:

Milestone What It Proves Key Risk Retired
M1: Local Room Prototype Can 4 people join a browser room, see/hear each other, and produce a composed 720p/30fps output? Browser canvas composition performance; SFU integration viability
M2: Broadcast Bridge Spike Can we take that composed output and deliver it to YouTube? WHIP → GStreamer → RTMP pipeline works end-to-end
M3: Product Shell Can we wrap the proven media path in a real product (accounts, persistent rooms, broadcast records)? None — product infra is known territory
The insight M1 and M2 are spikes — experiments to retire risk. If browser composition doesn't hit 28fps, or the WHIP-to-RTMP bridge doesn't work, you'd rather discover that before building auth, a database schema, and email flows. M3 is the "normal" product work that only happens once the hard media questions are answered.

Notice how M1 is authless and ephemeral — no database, no accounts, no persistence. That's deliberate (ADR 0020). You want to validate media capabilities without the overhead of product infrastructure slowing you down.
