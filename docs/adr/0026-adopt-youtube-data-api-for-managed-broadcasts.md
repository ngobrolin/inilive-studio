# Adopt the YouTube Data API for Managed Broadcasts

[ADR 0006](0006-use-pasted-youtube-stream-credentials-for-v1.md) deliberately kept v1 on pasted RTMP credentials so the bridge could be proven without Google app verification. That choice has a permanent ceiling: because the product never creates or transitions a YouTube `liveBroadcast`, YouTube treats the ingest as a bare stream key. The audience player does not show the red **LIVE** badge or sit at the live edge, latency mode and DVR cannot be controlled, and the Host must hand-manage scheduling, title, and visibility in YouTube Studio. These are the same gaps that make the output feel different from StreamYard, which drives the broadcast lifecycle through the API.

A future version should adopt the YouTube Data API v3 live-streaming endpoints to own the broadcast lifecycle: link a Host's YouTube channel through Google OAuth (`youtube` scope), create a `liveStream` (RTMP ingestion) and a `liveBroadcast` (the public event with title, visibility, latency, and `monitorStream`), bind them, and transition the broadcast through `testing → live → complete`. The existing server-side Broadcast Bridge ([ADR 0005](0005-use-a-server-side-broadcast-bridge.md), [ADR 0014](0014-run-the-broadcast-bridge-as-a-separate-process.md)) keeps doing exactly what it does today — WHIP ingest, H.264/AAC transcode, RTMP push — but now targets the API-issued ingestion address and stream key instead of pasted values. YouTube's archive remains the recording of record ([ADR 0008](0008-rely-on-youtube-archive-for-v1-recordings.md)).

This decision accepts the costs ADR 0006 deferred: a Google Cloud project, OAuth consent-screen verification for the `youtube` scope, refresh-token storage for linked channels, and Data API quota management. In exchange the product gains a genuine LIVE badge, programmatic latency/DVR control, automatic event creation, and a one-click go-live flow that no longer depends on the Host configuring YouTube Studio by hand. Pasted credentials may remain as a fallback path for non-YouTube RTMP destinations.
